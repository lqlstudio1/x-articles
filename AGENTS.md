# 项目协作规则

本仓库用于管理 X 文章选题、正文、配图、短版、发布说明和账号增长规则。

## 五个长期协作工件

根目录长期保留五个协作工件，分别承担不同职责：

1. `AGENTS.md`
   - 写长期规则。
   - 只记录会反复影响实现、审计边界、协作方式、验证标准和禁区的规则。
   - 不写一次性进度。

2. `progress.md`
   - 记项目进度。
   - 每次完成可交付切片、验证通过、发现重大边界或调整状态时更新。
   - 保留日期、范围、已完成项、验证结果和剩余风险。

3. `knowledge_map.md`
   - 画项目地图。
   - 记录目录结构、文章入口、配图脚本、账号规则、核心链路和文档索引。
   - 让新一轮工作可以先定位再修改。

4. `daily_task.md`
   - 推进任务。
   - 维护当前优先级、今日/下一步任务、阻塞项、验收标准和建议命令。
   - 用于把连续工作拆成小而可验证的切片。

5. `mistakes.md`
   - 记录错误和复测。
   - 所有已经暴露过的访问错误、菜单缺失、接口路径不匹配、数据初始化遗漏、测试失败和文档格式问题，都要记录现象、原因、修复、复测命令和当前状态。

## Git 提交规则

- 本仓库必须使用 Git 管理内容。
- 每次完成一篇文章、配图、脚本、规则更新或可交付切片后，都要提交到 Git。
- 提交前先检查 `git status --short`，确认没有误提交 `.codex/`、`.agents/`、`.obsidian/` 等本地状态目录。
- 提交信息要简短说明交付内容，例如 `Add AI song monetization article package` 或 `Update visual style rules`。
- 如果 Git 提交失败，必须在最终回复里说明失败原因和当前未提交文件。

## X 文章默认交付物

每次写 X 文章，默认输出：

- 正文 Markdown
- 140 字以内短版
- 5:2 标题图
- 1-3 张正文配图
- `image-guide.md` 配图说明
- 正文中明确标出每张图片插入位置
- `assets/` 图片目录
- `tools/` 生成脚本
- 根目录 `README.md` 索引更新

## 标题图封面锚点规则

后续生成标题图时，默认使用“信息密度 × 视觉锚点”的封面锚点系统。

- 先判断内容类型，再选模板：冲突型、数字型、截图型、情绪型、生活方式拼贴型。
- 每张标题图必须有一个主视觉锚点：超大标题、巨型数字、手机/截图、结果卡片、人物或改前改后对比。
- X 标题图优先 5:2：左侧 45%-60% 放主标题和副标题，右侧 40%-55% 放视觉锚点。
- 主标题必须短，优先 2-8 个汉字；超过 12 个字拆成主标题 + 副标题。
- 图片里最多 2-3 个文字区，不塞正文段落。
- 生成后要做缩略图判断：主标题和主锚点在 80px 宽缩略图里仍然可辨认。
- 不复制第三方示例图、Logo、水印或具体素材；不伪造平台数据、收益截图、二维码、私密聊天或真实品牌界面。

## 写作主线

账号内容默认按 `70/30` 执行：

- 70%：AI 编程 / 自动化
- 30%：AI 自媒体创作

最高优先级选题是两者结合：用 AI 编程做自媒体自动化工具、内容生产流水线、可运行作品。

## X 增长与账号策略规则

这部分是 @superfang119 的长期增长规则。写文章、做选题、起 Thread、设计配图和安排发布节奏时默认遵守。

### 基础资产

发内容前先保证账号基础清晰：

- Bio：说明你是谁、帮谁解决什么问题、为什么值得关注。
- Pinned post：优先置顶高价值 Thread，不置顶普通碎片帖。
- 头像和横幅：清晰、有识别度，横幅要服务账号定位。

当前账号定位：

> 用 AI 写代码、做内容、搭自动化，把想法变成能跑的作品。

主 niche：

- AI 工具实测
- AI 工作流
- 自动化落地
- AI 编程与内容生产
- Prompt -> 工作流 -> 可运行作品

### 内容优先级

后续选题优先级按下面执行：

1. AI 工具实测对比 Thread
   - 频率：每周 2-3 条。
   - 适合：Seedance、Codex、Claude Code、Cursor、n8n、Dify 等工具对比。
   - 必须有相同输入、结果截图/视频、成本/速度/质量/适用场景对比。

2. 完整工作流 Thread
   - 频率：每周 1-2 条。
   - 适合：Codex / Claude Code / 自动化 / 内容生产流水线。
   - 必须讲清输入、工具组合、步骤、踩坑、输出和复用方式。

3. Prompt + 案例拆解
   - Prompt 不能孤立发布。
   - 必须配结果、失败版本、改进逻辑或应用场景。

4. 想法 -> 可运行作品案例
   - 最符合账号 Bio。
   - 必须展示最终成果，并拆出“人负责判断，AI 负责执行”的边界。

5. 单帖实测 + 视频 / 截图
   - 用于日常更新。
   - 必须有真实观察，不发空泛工具资讯。

通用内容类型可以借鉴 8 类：

- Observation：观察分享
- Explainer：拆解概念或事件
- Personal story：个人经历和教训
- How to：步骤清晰的方法
- X vs Y：工具、方法或观点对比
- Contrarian：反直觉观点和理由
- Listicle：编号清单
- Meme / 幽默观察：少量使用，服务人格化

### 双主线比例

默认内容比例：

```text
70% AI 编程 / 自动化
30% AI 自媒体创作
```

优先写两者结合的内容：

- 用 AI 编程做自媒体自动化工具。
- 用 Codex 开发标题图、短版文案、发布清单生成器。
- 用 n8n + Dify + Codex 搭内容生产流水线。
- 把“写文章 -> 配图 -> 短版 -> 复盘”做成半自动流程。

### Thread 结构

Thread 建议 5-12 条。默认结构：

```text
Hook -> 问题/误区 -> 框架 -> 步骤/案例 -> 踩坑 -> 结论 -> 具体 CTA
```

第一条只负责让人停下来。优先使用：

- 数字
- 对比
- 问题
- 反直觉判断
- 真实结果

中间必须交付真实价值：

- 可执行步骤
- 数据或截图
- Prompt 或流程
- 真实案例
- 工具对比
- 失败经验

最后 CTA 优先写具体问题，不要只写“关注我”。例如：

- 你更想看完整 Prompt，还是完整工作流？
- 你最想自动化哪一步？
- 你还想看哪个工具对比？

### 三个固定 Thread 模板

#### 模板 1：工具对比类

```text
Hook：
相同 Prompt 下，A 和 B 到底差多少？

结构：
1. 测试目标
2. 相同 Prompt / 输入
3. A 的结果 + 截图/视频
4. B 的结果 + 截图/视频
5. 成本、速度、质量、文件大小或积分消耗
6. 我的结论：什么时候用 A，什么时候用 B

CTA：
你还想看哪个工具用同一任务对比？
```

#### 模板 2：Codex 工作流类

```text
Hook：
很多人把 Codex 当 ChatGPT 用，浪费了大部分能力。
正确用法是：把它当成一个需要管理的 AI 同事。

结构：
1. 常见错误：直接让它写代码
2. 读项目结构
3. 输出开发计划
4. 小步迭代 + 跑测试
5. Review diff + 留评论
6. 沉淀到 AGENTS.md / Skills
7. 自动化后续任务
8. 实际案例截图或运行结果

CTA：
你更想看 Codex 工作流截图，还是 AGENTS.md / Skills 示例？
```

#### 模板 3：案例复盘类

```text
Hook：
我用 Claude Code + Codex，把一个想法做成了可运行项目。

结构：
1. 原始想法
2. 为什么值得做
3. 如何拆任务
4. AI 负责什么，人负责什么
5. 最终成果截图/视频/源码思路
6. 中间最大问题
7. 下次复用这套流程的方法

CTA：
你有哪个想法想变成可运行作品？
```

### 发帖节奏

小号阶段不要机械狂发，目标是稳定形成识别度。

建议节奏：

- 每天 1-2 条高质量内容。
- 每周至少 2-3 条结构化 Thread。
- 每周至少 1 条工具实测对比。
- 每周至少 1 条完整工作流或案例复盘。
- 每天 20-40 分钟战略回复。

中文用户优先发布时间：

- 08:00-10:00
- 12:00-14:00
- 20:00-22:00

### 战略回复

回复是小号阶段的重要分发方式，但不能做批量刷回复。

优先回复：

- AI 工具测评账号
- AI Coding / Cursor / Claude / Codex 相关账号
- AI 视频生成账号
- 内容创作 / 自动化 / No-code 账号
- 粉丝量约 5k-50k 的垂直账号

高质量回复必须提供增量：

- 实测细节
- 参数或步骤
- 失败案例
- 风险边界
- 不同场景
- 后续问题

示例：

```text
这个工具我也测过。我的感觉是 [具体场景] 下 A 更稳，但 [另一个场景] B 更合适。尤其是 [参数/步骤] 会明显影响结果。
```

```text
Codex 我现在更偏“读项目 -> 出计划 -> 小步迭代 -> 跑测试 -> Review diff”这套，比直接让它一口气写代码稳定很多。
```

### 置顶策略

Pinned post 优先换成强 Thread：

1. Codex 工作流 Thread
2. AI 工具实测对比 Thread
3. 想法 -> 可运行作品案例 Thread

置顶帖必须包含：

- 账号主张
- 可复制流程
- 结果截图或视频
- 明确 CTA

### 复盘指标

发布后不要只看浏览量。优先记录：

- 是否带来 profile visit
- 是否带来 follow
- 是否有收藏
- 是否有高质量评论
- 是否有人引用观点
- 哪张图提高了点击
- 哪类 Hook 更容易引发回复

浏览量是噪音，关注理由才是资产。

## 验证标准

文章类交付完成前至少检查：

- 正文是否有明确判断，而不是泛泛信息搬运。
- 是否归类到工具对比、工作流拆解、案例复盘、方法清单、反直觉观点或战略回复延展。
- 是否有标题图和正文配图。
- 正文是否已经标出标题图和每张正文配图的插入位置。
- 图片中文字是否清晰、手机缩略图是否能看懂。
- `README.md` 是否登记文章、配图、说明和脚本。

## 禁区

- 不写收益保证、稳赚、无脑赚钱。
- 不把未经证实的平台算法说法写成确定事实。
- 不鼓励刷量、互赞互粉、批量回复或规避平台规则。
- 不把别人的截图、文章或素材直接当作原创配图。
- 不只做信息转述，必须加入判断、验证、流程或案例。

## 绘图执行补充

后续生成标题图、封面图和正文配图时，默认使用 `references/visual-style/ponyo-cover-anchor-drawing-rules.md` 的绘图规则。

- 默认生成完整成品图，不只生成空背景，除非用户明确要求后期叠字。
- 每张图先确定模板类型、一眼承诺、视觉锚点、锚点位置和文字层级。
- 图片不能只堆文字；除纯信息图外，标题图和正文图都要尽量结合文章内容加入场景化视觉元素，如人物、物体、设备、桌面、手机、代码界面、工具界面、数据面板、工作现场或生活场景。
- X 文章标题图默认 5:2，左侧 45%-60% 放超大标题，右侧 40%-55% 放卡片、截图、数字、手机框或结果面板。
- 小红书 / 生活方式封面才优先使用 3:4、涂鸦描边小清新或阳光胶片拼贴白描边风。
- 所有图都要通过 80px 缩略图可读性判断：主标题和主锚点仍能识别。
- 不复制第三方仓库示例图、Logo、水印或具体素材；只吸收方法论和布局规则。

## 视觉风格轮换规则

后续生成标题图和封面图时，不能机械套同一种视觉风格。除非用户明确指定，默认先参考 `references/visual-style/multi-cover-style-board-notes.md` 选择风格。

- AI 编程 / 工具测评：优先霓虹科技、手机截图测评、数据结果面板、粗字冲击。
- 自媒体创作 / 封面方法：优先改前 VS 改后、巨型数字、纸感清单、评论截图。
- 个人复盘 / 焦虑 / 长期主义：优先情绪人像、夜间生活方式、治愈极简。
- 平台规则 / 风险提醒：优先冲突人像、粗字冲击、数据面板。
- 连续 3 篇文章的标题图尽量不要使用同一种模板。
- 正文解释图优先清晰，不必为了风格变化牺牲可读性。
- 每篇文章至少一张图应有“场景感”或“实物感”，不要全部做成文字卡片、流程框或纯排版海报。
