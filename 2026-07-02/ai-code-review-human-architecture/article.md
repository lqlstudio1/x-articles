# AI 写的代码，到底要不要人工审？

## 发布版

我的结论很简单：

AI 生成的代码当然要审。

但人不应该再用过去那种方式审：一行一行看语法、变量名、注释、logging、边界条件。

这些事应该交给测试、规范和 AI code review。

人真正要审的，是 AI 最容易写坏、但表面又最像“没问题”的东西：

```text
架构设计、模块边界、状态流向、长期维护成本。
```

## 1. AI 代码最大的风险，不是写错，而是写“看起来很对”

过去人工 review 的很多工作，本质是在查局部问题：

- 这里会不会空指针？
- 这个异常有没有处理？
- 这个变量名是否清楚？
- 日志有没有打？
- 单测有没有覆盖？
- 是否符合团队规范？

这些问题还重要，但它们越来越不该成为人的主要工作。

因为在 agentic coding 时代，AI 可以写代码，也可以写测试，还可以按规则做 code review。

只要团队有清晰规范，AI review 能吃掉大量过去靠肉眼检查的内容。

真正麻烦的是另一类问题：

```text
每一行代码都没错，但整个设计是错的。
```

比如一个简单功能，被 AI 拆成 7 个无意义的 helper。

比如本来应该复用已有服务，它重新造了一套平行逻辑。

比如状态被塞进了错误的层，短期能跑，三个月后没人敢改。

这种代码最危险，因为它通常能通过测试，也符合格式，甚至 review 摘要看起来很漂亮。

## 2. 自动化测试要升到第一优先级

AI 写代码越快，测试越重要。

不是因为测试能证明代码完美，而是因为测试能把“可验证行为”从人工审阅里剥离出去。

凡是能自动验证的行为，都应该自动验证。

这包括：

- 单元测试；
- API 行为；
- 权限边界；
- 表单校验；
- 数据迁移；
- 关键页面流程；
- 错误状态；
- 回归用例。

AI 写单元测试已经很容易。

但复杂集成测试、端到端测试、真实业务链路测试，仍然需要人去设计测试边界。

这里人的价值不是“亲手写每一行测试代码”，而是定义：

```text
什么行为必须被保护？
哪些路径一旦错了会造成真实损失？
哪些用例值得长期维护？
```

换句话说，人要负责测试策略，AI 负责补齐测试实现。

## 3. AI review 可以审规范，人类 review 应该审设计

我更倾向把 review 分成三层。

第一层，机器检查：

- lint；
- type check；
- formatting；
- test；
- build；
- security scan。

第二层，AI review：

- 命名是否清楚；
- 注释是否多余；
- 边界条件是否遗漏；
- error handling 是否一致；
- logging 是否合适；
- 是否违反项目约定；
- 是否有重复代码；
- 是否有明显性能问题。

第三层，人工 review：

- 这个模块该不该存在？
- 这个抽象是不是过早？
- 状态放在这里是否合理？
- 是否破坏已有业务边界？
- 以后加新需求会不会更难？
- 这次修改有没有引入新的长期维护成本？

前两层追求“有没有错”。

第三层追求“这个设计以后能不能活”。

这才是人工审阅在 AI 时代最该保留的位置。

## 4. 不要让人读巨量代码，要让 AI 提炼设计变化

AI 一次能生成几百行、几千行代码。

如果还要求人从头读到尾，review 会变成假动作。

更好的方式是让 AI 在每次提交里补充三类信息。

第一类，改了什么：

```text
- 新增了哪些文件
- 修改了哪些模块
- 删除了哪些旧逻辑
- 哪些接口或数据结构发生变化
```

第二类，为什么这么改：

```text
- 这个方案解决什么问题
- 为什么没有复用旧逻辑
- 为什么新增抽象
- 有哪些替代方案被放弃
```

第三类，架构影响：

```text
- 数据从哪里来，到哪里去
- 状态由谁持有
- 失败时怎么回滚
- 未来扩展点在哪里
- 哪些地方需要重点 review
```

人不需要先看代码。

人应该先看这份摘要和架构图。

如果摘要讲不清，说明这次修改本身就有问题。

如果架构图画不出来，说明 AI 可能只是堆代码，不是真的理解系统。

## 5. 哪些代码可以不人工 review？

不是所有 AI 代码都值得人工审。

这几类场景，可以降低人工 review 强度：

- 一次性脚本；
- 数据清洗脚本；
- 小型内部工具；
- 成熟框架里的常规 CRUD；
- 低风险页面样式调整；
- 有完整测试覆盖的机械重构。

但下面这些场景，必须保留人工 review：

- 长期维护项目；
- 多人协作项目；
- 核心业务链路；
- 权限、支付、账号、数据安全相关代码；
- 跨模块状态变更；
- 新增基础抽象；
- 改数据库结构；
- 改接口契约；
- 改部署和运维链路。

判断标准不是“AI 写了多少代码”。

判断标准是：

```text
这次修改如果设计错了，未来修复成本有多高？
```

成本越高，越需要人审。

## 6. 最值得维护的不是规范文档，而是设计记忆

如果团队真要长期使用 AI 写代码，最好维护一份项目设计文档。

它不需要很复杂，但至少要记录：

- 项目目录结构；
- 核心模块职责；
- 前后端入口；
- 数据库脚本；
- 关键接口；
- 主要状态流；
- 不要碰的边界；
- 常见错误；
- 已做过的设计取舍。

这份文档要让 AI 每次修改后主动更新。

因为它同时服务两类读者：

```text
人：快速判断这次改动有没有破坏设计。
AI：下次接任务时少走弯路，少浪费上下文。
```

没有设计记忆，AI 每次都是“第一次进项目”。

有了设计记忆，AI 才可能从写代码，进化成参与维护系统。

## 结论

AI 代码要不要人工审？

要。

但人工 review 的重点必须变化。

不要把人的时间浪费在 AI、lint、type check、test 都能处理的问题上。

人的价值应该放在三件事上：

1. 定义自动化测试保护哪些行为；
2. 判断架构设计和模块边界是否合理；
3. 维护项目设计记忆，让 AI 下次少犯同样的错。

AI 写代码之后，人类不是代码检查员。

人类更像技术负责人：

```text
不盯每一行，而是盯方向、边界和长期成本。
```

这才是 agentic coding 时代真正有效的 code review。
